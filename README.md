# ⚙️ Limit Orderbook Simulator

Симулятор биржевого стакана заявок (Limit Order Book) на C++20 - с движком сведения заявок в сделки, несколькими моделями генерации потока заявок и интерактивным веб-дашбордом для наблюдения за поведением смоделированного рынка в реальном времени.

Проект построен по мотивам книги *"Trades, Quotes and Prices"* (Bouchaud, Bonart, Donier, Gould) - одной из ключевых монографий по микроструктуре рынка. Идея простая: смоделировать стакан заявок с нуля, наполнить его реалистичным потоком случайных заявок (покупки, продажи, отмены) и проверить на данных закономерности из книги - например, что цена крупной покупки растёт не пропорционально объёму, а как квадратный корень от него.

Это не production-система для реальной торговли, а исследовательский и обучающий проект: engine книги заявок написан на C++ ради производительности, а вся логика доступна и настраиваема через веб-интерфейс, чтобы можно было визуально наблюдать, как разные параметры рынка влияют на цену, ликвидность и поведение книги заявок.

## ⚙️ Архитектура

Проект состоит из трёх слоёв, каждый со своей зоной ответственности:

### ![C++](https://img.shields.io/badge/C%2B%2B-20-blue) core

Самый нижний, самый производительный слой - здесь находится вся логика, критичная к скорости:

- **`OrderBook`** - движок сведения заявок в сделки (matching engine). Хранит книгу на `std::map<int, PriceLevel>` для каждой стороны (bid/ask), где каждый уровень цены — это `std::deque<Order>`, реализующий очередь по правилу price-time priority (кто раньше пришёл на данной цене — тот раньше исполнится). Отдельный `std::unordered_map` хранит, на каком уровне цены лежит заявка с конкретным ID, что даёт мгновенную (O(1)) отмену без перебора всей книги.

- **`OrderFlowGenerator`** - общий интерфейс для моделей потока заявок. От него наследуются:
  - **`SantaFeFlow`** - классическая zero-intelligence модель: три независимых пуассоновских потока (лимитные заявки, маркет-ордера, отмены), плюс гамма-распределение расстояния новой лимитки от текущей середины спреда (даёт реалистичную, колоколообразную форму глубины книги вместо равномерной).
  - **`HawkesFlow`** - self-exciting процесс (алгоритм прореживания Огаты): каждое новое событие немного повышает вероятность скорого следующего события, создавая наблюдаемую кластеризацию активности во времени.
  - Обе модели поддерживают DAR-модель (autoregressive) памяти знака сделок - параметр `ρ`, задающий вероятность, что новый маркет-ордер повторит знак предыдущего.
- **`MetaorderExecutor`** - механизм дробления крупной заявки на много мелких child-ордеров, исполняемых постепенно, вперемешку с фоновым потоком — используется для проверки square-root law price impact.

### О выборе типа данных для цены

Цена в ордербуке хранится как `int`, а не `float`/`double`. Это осознанное решение, а не упрощение: такой подход избавляет от ошибок округления, которые неизбежно возникают при сравнении и суммировании чисел с плавающей точкой.

### ![Python](https://img.shields.io/badge/Python-3.11-green) связующий слой

C++ классы напрямую собираются в Python-модуль через **pybind11** — без сети и сериализации между Python и C++, вызов метода `OrderBook` из Python эквивалентен прямому вызову C++ функции в памяти процесса. Замеры показали, что сама C++ логика (matching, генерация событий) занимает <1% от полного времени HTTP-запроса — основная задержка приходится на сеть и сериализацию, а не на вычисления.

### ![FastAPI](https://img.shields.io/badge/FastAPI-009688?logo=fastapi&logoColor=white) backend

REST API поверх pybind11-модуля:
- `/step` - продвигает симуляцию на N событий, отдаёт текущее состояние книги и live-метрики
- `/params` - на лету пересоздаёт генератор потока с новыми параметрами (λ, μ, ν, φ₀, g, ω, ρ и др.)
- `/metaorder/start` + `/metaorder/status` - запуск и пошаговое отслеживание исполнения метаордера
- `/autocorrelation` - считает автокорреляционную функцию знака исполненных сделок по накопленной истории

### ![React](https://img.shields.io/badge/React-18-61DAFB)  frontend

Дашборд на React (сборка через Vite, графики через Recharts): живая визуализация глубины книги, mid-price, спреда, ленты потока заявок, impact path метаордера, scatter-графика square-root law и графика автокорреляции знака сделок - все параметры модели настраиваются слайдерами без перезапуска приложения.

### ![Docker](https://img.shields.io/badge/Docker-2496ED?logo=docker&logoColor=white) Деплой

Всё оборачивается в **Docker Compose** - backend (Python + скомпилированный C++ модуль) и frontend поднимаются одной командой, C++ компилируется прямо при сборке образа.

## ⚙️ Реализованный функционал

**Движок L2-ордербука**
- Полноценный matching engine с приоритетом цена-время (price-time priority / FIFO)
- Поддержка лимитных заявок, маркет-ордеров и отмен
- Крупный ордер может исполниться сразу по нескольким уровням цены.
- Пропускная способность ~439 000 событий в секунду в однопоточном режиме (замерено через throughput_test.cpp)

**Генерация потока заявок**
- Две сменяемые модели: Santa Fe (случайный поток без памяти) и Hawkes (поток с самовозбуждением - события кластеризуются во времени)
- Реалистичное, колоколообразное распределение объёма в книге за счёт гамма-распределения расстояния новой заявки от середины спреда
- DAR-модель памяти знака сделок — можно включить эффект, при котором сделки одного направления идут сериями, как это происходит в реальности при исполнении крупных заказов

**Метаордеры и влияние на цену**
- Возможность отправить крупную заявку, которая исполняется постепенно, маленькими частями, вперемешку с обычным потоком
- Автоматический расчёт цены до и после исполнения, построение графика движения цены во время исполнения
- Отдельный скрипт для массового тестирования: подтверждена корневая зависимость влияния объёма на цену (square-root law) с R² = 0.996 на 440 симуляциях, а также найдена граница, где эта зависимость перестаёт работать — когда объём заявки становится сопоставим с объёмом всей книги

**Дашборд и метрики**
- Живая визуализация книги заявок: таблица, график глубины, профиль объёма.
- График цены (mid-price) и спреда во времени
- Индикатор дисбаланса книги (перевес покупателей или продавцов)
- График автокорреляции знака сделок - показывает, насколько предсказуемо направление следующей сделки
- Разбивка задержки запроса по этапам: вычисления, сериализация данных, сеть

- REST API на FastAPI
- Frontend на React
- Полный запуск одной командой через Docker Compose


## ⚙️ Ключевые результаты

**Производительность движка**
Matching engine на C++20 обрабатывает **~439 000 событий в секунду** в однопоточном режиме (смешанный поток лимитных заявок, маркет-ордеров и отмен). Для сравнения: полный HTTP-запрос от браузера до сервера и обратно занимает ~15 мс, из которых на саму C++ логику уходит меньше 1% времени — основная задержка приходится на сеть и сериализацию данных, а не на вычисления.

**Square-root law price impact**
На отдельном скрипте (`sq_root_law_test.py`) прогнано 440 независимых симуляций метаордеров разного объёма (от 50 до 3000 единиц), с усреднением по 40 повторам на каждый размер. Результат подтверждает классическую эмпирическую зависимость из финансовой литературы: влияние крупной заявки на цену растёт не пропорционально объёму, а как квадратный корень от него.

```
impact ≈ Y · √Q,   Y = 0.10969,   R² = 0.9963
```

Также обнаружена естественная граница применимости закона: при объёме заявки, сопоставимом с объёмом всей доступной ликвидности книги, зависимость перестаёт выполняться — метаордер начинает "не помещаться" в книгу и полностью её опустошать.

**Автокорреляция знака сделок**
Проведён факторный эксперимент : две модели потока заявок (Santa Fe с λ=2, μ=1, ν=0.5; Hawkes с φ₀=1, g=0.7, ω=0.5) на трёх значениях параметра памяти знака DAR (ρ = 0, 0.5, 1).

При ρ = 0 обе модели дают автокорреляцию, колеблющуюся в пределах шума вокруг нуля — при заданных параметрах ни случайный поток (Santa Fe), ни самовозбуждающийся поток (Hawkes) сами по себе не создают выраженной памяти знака сделок.

При включении DAR (ρ = 0.5) в обеих моделях появляется практически идентичная, плавно затухающая положительная автокорреляция (~0.5 на малых лагах, спадающая к нулю к лагу ~10) — то есть память знака при этих параметрах определяется исключительно явным DAR-механизмом, а не выбором модели потока.

При ρ = 1 (знак market order всегда повторяет предыдущий) автокорреляция выходит на максимум (~0.85–1.0) и медленно спадает на всём диапазоне лагов — ожидаемое поведение для предельного случая.

Общий вывод: в текущей конфигурации параметров DAR — единственный источник наблюдаемой долгой памяти знака сделок, а не побочный эффект временной кластеризации Hawkes. Это не противоречит теории (глава 10 книги называет order-splitting/DAR основным механизмом).

## ⚙️ Запуск

Нужен установленный [Docker](https://www.docker.com/) и Docker Compose.

```bash
git clone https://github.com/IvanVereshchagin/lob-simulator.git
cd lob-simulator
docker-compose up --build
```

После сборки (C++ модуль компилируется прямо внутри контейнера, это может занять пару минут при первом запуске):

- Backend (FastAPI): `http://localhost:8000`
- Frontend (дашборд): `http://localhost:5173`

Документация REST API доступна по адресу `http://localhost:8000/docs` (автоматическая Swagger-документация FastAPI).

### Отдельный скрипт для проверки square-root law

Скрипт `sq_root_law_test.py` запускает серию независимых симуляций и печатает результат регрессии в консоль — не требует запущенного backend/frontend, использует C++ модуль напрямую через pybind11:

```bash
python sq_root_law_test.py
```

## ⚙️ Методология

Проект опирается на книгу *"Trades, Quotes and Prices: Financial Markets Under the Microscope"* (Jean-Philippe Bouchaud, Julius Bonart, Jonathan Donier, Martin Gould, Cambridge University Press) — одну из наиболее полных современных монографий по микроструктуре финансовых рынков. В частности, реализованы и проверены следующие концепции из книги:

- **Price-time priority matching** и механика L2 ордербука (глава 3)
- **Santa Fe zero-intelligence model** — базовая модель потока заявок без стратегического поведения участников (глава 8)
- **Hawkes-процессы** для моделирования кластеризации событий во времени (глава 9)
- **Долгая память знака сделок** и её связь с дроблением крупных заявок (глава 10)
- **Метаордеры и square-root law price impact** — эмпирическая зависимость влияния объёма на цену (глава 12)

Книга не даёт готового кода — вся реализация, архитектурные решения и эксперименты по проверке закономерностей на синтетических данных сделаны самостоятельно, с использованием теоретических моделей и эмпирических результатов книги как ориентира.
